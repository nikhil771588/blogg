Database Recovery Techniques

Database recovery techniques ensure that data remains consistent and available even after failures. Below are some key techniques:

1. Log-Based Recovery

Uses transaction logs to restore data after crashes.

Log-based recovery is a technique used in database management systems (DBMS) to ensure data integrity and consistency after a failure.
This method relies on maintaining a log file that records all database transactions before they are committed. In case of a crash or failure, the system uses these logs to restore the database to a consistent state.

1. Types of Logging in DBMS

Log-based recovery involves storing logs that help in rolling back uncommitted transactions or redoing committed ones. The common types of logging include:
Redo Logging (Deferred Update)
Changes are recorded in the log before they are written to the database.
Transactions are not applied to the database until they are fully committed.
Recovery process involves redoing committed transactions from the log.
Undo Logging (Immediate Update)
Database updates occur immediately, but logs allow for rolling back changes if needed.
Uncommitted transactions can be undone using the recorded log entries.
Undo/Redo Logging (Combination Approach)
Logs contain both undo and redo information.
Allows for flexible recovery by rolling back uncommitted transactions and reapplying committed ones.

2. Log-Based Recovery Process

The recovery process involves two main steps:
a) Undo Phase
Used to roll back uncommitted transactions.
Transactions that started but were not completed before the crash are undone using the log.
b) Redo Phase
Used to reapply committed transactions.
Ensures that all committed transactions are reflected in the database.
3. Checkpointing in Log-Based Recovery
 To optimize recovery, checkpoints are periodically created in the log. A checkpoint saves the current state of the database, reducing the amount of log that needs to be processed during recovery.
Consistent Checkpointing: Ensures that all transactions before a checkpoint are completed.
Fuzzy Checkpointing: Allows transactions to be active during checkpoint creation.
4. Advantages of Log-Based Recovery
 Ensures data consistency even after system failures.
Supports automatic recovery with minimal manual intervention.
Reduces data loss by recording every transaction.
Works efficiently with concurrent transactions.

5. Limitations of Log-Based Recovery
 Storage overhead: Logs require additional storage space.
Performance impact: Writing logs before committing transactions may slow down operations.
Complex recovery: Managing redo and undo processes can be complex in large-scale systems.
